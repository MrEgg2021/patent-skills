# 专利说明书核心发明点解析提示词（研发阅读版）

## 任务

你将接收一个或多个仅包含专利说明书正文的文本。请为每篇说明书提取少量核心发明点，并以 JSON 输出。

目标读者是研发人员。输出应帮助研发人员快速理解该专利解决什么技术问题、采用什么关键技术方案、哪些结构/步骤/算法/参数/协同关系值得关注。

## 核心原则

1. **严格来源于说明书**：所有技术问题、技术方案、参数、效果、材料、模型结构、步骤关系都必须来自说明书原文。
2. **禁止幻觉**：原文没有的数值、比例、效果、实验结论、部件关系、算法细节，一律不得补。
3. **少量但不极端压缩**：通常提取 2-5 个核心发明点；确属单一发明构思时可输出 1 个；复杂组合方案可超过 5 个，但必须合并同质细节。
4. **研发可读**：技术方案不能只有抽象标题，应说明关键实现机制。避免把整段说明书机械复制成流水账。
5. **一一对应**：`技术问题` 数组与 `技术方案` 数组必须同序号对应。

## 解析方法

### 1. 识别技术问题

优先从“背景技术”“发明内容”中识别现有技术缺陷、技术瓶颈、待解决问题。

每个技术问题应满足：
- 指向具体技术缺陷或挑战，而不是商业价值口号；
- 保留原文中明确出现且与问题直接相关的量化指标；
- 原文没有量化指标时，不自行添加；
- 表述长度以研发可理解为准，通常 1-3 句。

### 2. 识别技术方案

从“发明内容”“具体实施方式”“附图说明”中提取解决对应问题的核心技术手段。

每个技术方案应包含足够让研发理解的关键信息，例如：
- 机械/结构：关键部件、连接关系、运动关系、配合方式；
- 电学/控制：模块构成、信号流、控制逻辑、硬件支撑；
- 算法/软件/AI：输入、处理步骤、模型/规则/算法改进、输出；
- 工艺/方法：关键步骤、顺序、参数条件、设备/原料；
- 跨领域方案：不同技术模块之间如何协同。

技术方案通常 2-5 句。可以概括整合，但不得加入说明书没有的信息。

### 3. 过滤常规内容

以下内容通常不单独作为发明点，除非说明书明确把它作为创新或必要实现机制：
- 通用标准组件；
- 常规数据读取、存储、显示、传输；
- 普通外壳、支架、电源、处理器等无特殊限定部件；
- 与解决技术问题无直接关系的实施例细节。

### 4. 多发明点合并与拆分

- 若多个实施例服务同一技术问题且采用同一核心机制，合并为一个发明点。
- 若同一专利中存在相互独立的问题-方案闭环，应拆成多个发明点。
- 若一个总方案下有关键子机制，且研发理解时必须单独掌握，可拆成独立发明点。

## 输出格式

直接输出 JSON 数组，不输出解释文字。每个对象代表一篇专利：

```json
[
  {
    "专利公开号": "从文件名或文本中提取；无法识别则填文件名",
    "专利标题": "说明书标题；无法识别则填（未识别）",
    "技术问题": [
      "1个技术问题，对应技术方案数组第1项",
      "另1个技术问题，对应技术方案数组第2项"
    ],
    "技术方案": [
      "解决第1个技术问题的核心技术方案",
      "解决第2个技术问题的核心技术方案"
    ]
  }
]
```

## 失败和不确定处理

- 无法识别标题：`"专利标题": "（未识别）"`。
- 无法识别公开号：用文件名或输入标识代替。
- 某篇说明书无法解析：保留该篇对象，并在 `技术问题` / `技术方案` 中写明 `解析失败：原因`。
- 章节截取可疑时，不要假装确定；在对应字段中保守表述，避免引入原文外信息。

## 禁止事项

- 不要输出 Markdown 解释、任务复述或分析过程。
- 不要输出旧字段 `"文件名称"` / `"发明点"`，除非调用方明确要求兼容旧格式。
- 不要为了凑数量强行拆分。
- 不要把摘要或权利要求内容作为发明点来源。
