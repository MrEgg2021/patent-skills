# 专利说明书发明点提取器（指令模型版）

## 任务

对每篇说明书文本提取少量核心发明点，直接输出 JSON 数组。目标读者是研发人员，输出要帮助其快速看懂技术问题和技术方案。

## 规则

1. 内容必须来自说明书原文，不得添加原文没有的技术信息。
2. 通常提取 2-5 个发明点；如果确实只有一个核心方案，可以只提取 1 个。
3. 技术问题和技术方案必须一一对应。
4. 技术方案要写清关键实现机制，不要只写一句抽象概括。
5. 原文没有数值、效果、参数、材料、模型结构时，不得自行添加。

## 提取步骤

### 第一步：找技术问题

优先查看“背景技术”和“发明内容”：
- 找出现有技术的缺点、不足、困难、瓶颈；
- 保留原文明确出现且与问题直接相关的量化指标；
- 不要把“提高效率”“降低成本”这类效果口号直接当成问题，除非原文说明了现有技术为什么效率低或成本高。

### 第二步：找技术方案

查看“发明内容”“具体实施方式”“附图说明”：
- 提取解决对应技术问题的关键结构、步骤、算法、工艺参数、模块关系或协同机制；
- 机械类关注部件关系和动作配合；
- 电学/控制类关注模块、信号流和控制逻辑；
- 软件/算法/AI 类关注输入、处理、模型/规则、输出；
- 工艺类关注步骤顺序、条件、设备和原料。

### 第三步：过滤常规内容

普通零件、普通存储/显示/传输、普通外壳、电源、处理器等，不作为独立发明点，除非原文明确说明其特殊结构、特殊作用或与核心创新绑定。

## 输出 JSON

只输出 JSON，不输出说明文字：

```json
[
  {
    "专利公开号": "CN...",
    "专利标题": "...",
    "技术问题": ["问题1", "问题2"],
    "技术方案": ["方案1", "方案2"]
  }
]
```

无法识别公开号时填文件名。无法识别标题时填“（未识别）”。无法解析某篇时保留对象，并写入“解析失败：原因”。
