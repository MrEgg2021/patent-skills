# 权利要求技术特征拆解提示词

## 任务

根据用户提供的权利要求书和说明书正文，将每项权利要求拆解为可独立阅读和比对的技术特征，并匹配说明书中的对应支持内容。最终输出可直接转 xlsx 的 JSON 数组。

## 输入

用户应提供：
1. 权利要求书全文；
2. 说明书正文，通常包括技术领域、背景技术、发明内容、附图说明、具体实施方式，不含摘要；
3. 如文本章节不完整，应按已提供内容保守分析。

## 总原则

1. **权利要求原文是证据基础**：技术特征列必须以权利要求原文为主体。
2. **不擅自改写权利要求**：不得把权利要求原文改成自己的技术概括，不得删除限定词。
3. **允许极少量语义标注**：当单个特征因“其”“所述”“该”等指代或省略而无法独立理解时，可在原文后追加极短标注，例如 `（标注：其中“其”指代支撑架）`。标注必须可删除，不得替代原文。
4. **从属权利要求只拆新增限定**：不要重复被引用权利要求中已有的特征。
5. **说明书描述严格溯源**：说明书没有明确描述的内容，不得推测或补充。
6. **效果与优势优先取本发明描述**：背景技术可用于理解问题，但不得把背景技术中的缺陷直接包装成该特征的技术效果。

## 拆解规则

### 权利要求序号

- 独立权利要求填写数字，如 `"1"`。
- 从属权利要求填写直接引用关系，如 `"2 (引1)"`、`"5 (引1, 3)"`。
- 同一权利要求拆出多条技术特征时，每条均重复填写该序号。

### 技术特征

拆解粒度以“语义单位独立且可比对”为准：
- 并列部件、并列步骤、并列模块通常拆开；
- 一个部件的必要结构限定可与该部件合并，避免过碎；
- 方法步骤按实际时序或逻辑处理单元拆分；
- 选择性/马库什表述可整体作为一个特征，并在说明书描述中展开各选项；
- 从属权利要求只记录其新增限定。

技术特征列写法：

```text
权利要求原文
```

必要时：

```text
权利要求原文（标注：其中“其”指代XXX）
```

不得写成：

```text
模型概括后的技术点
```

### 说明书描述

围绕该技术特征，在说明书中查找并整合以下内容：
- 工作原理/作用机制；
- 具体结构、组成、步骤或实施例；
- 参数、范围、条件、材料、模型结构；
- 连接、位置、时序、交互关系；
- 技术效果或优势；
- 对整体技术问题的贡献。

每一类内容都必须标注来源，例如：

```text
工作原理：...（来源：【0031】-【0033】）
具体实施例：...（来源：【0040】，参见图2）
```

若没有找到任何对应描述，填写：

```text
说明书中未明确描述
```

## 输出格式

最终只输出 JSON 数组，可包裹在 ```json 代码块中。数组中每个对象代表表格一行，键名必须完全一致：

```json
[
  {
    "权利要求序号": "1",
    "技术特征 (源自权利要求书原文，按语义单位独立分解)": "权利要求原文（标注：必要时极少量指代说明）",
    "说明书描述 (主题归类式整合，严格源自说明书原文，并注明来源)": "工作原理：...（来源：【0031】）"
  }
]
```

## 输出前自检

输出前逐项检查：
1. 是否覆盖所有权利要求；
2. 从属权利要求是否只拆新增限定；
3. 技术特征列是否仍以原文为主体；
4. 标注是否极少且明确为“标注”；
5. 说明书描述是否有来源；
6. 没有支持的特征是否明确写“说明书中未明确描述”；
7. JSON 是否可解析，键名是否完全一致。
